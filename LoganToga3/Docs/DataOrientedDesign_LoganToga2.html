LoganToga2 Data-Oriented Redesign

A proposed data-oriented architecture for the Siv3D/C++ LoganToga2 project that preserves existing gameplay requirements while improving cache locality, testability, save compatibility, and future content iteration.

Project: LoganToga2 Engine: Siv3D v0.6.15 Language: C++ Approach: Data-Oriented Design

1. Executive Summary

The current design stores most unit state and behavior in large object-style classes such as Unit, Skill, ClassBattle, ClassMapBattle, and CommonConfig. This is easy to extend locally, but costly for large battles because update systems repeatedly traverse arrays of heavy objects containing unrelated fields, strings, UI state, build state, combat state, movement state, and optional components.

The redesigned model separates stable definitions from runtime instances, stores hot runtime state in structure-of-arrays containers, and moves behavior into systems that operate on compact component stores. The goal is not to remove all classes, but to make the frequently updated battle data dense, explicit, and predictable.

Preservation rule: unit stats, skills, formations, buildings, resources, battle commands, map metadata, carrier behavior, save data, command-line entry points, and the current game flow are preserved. The redesign changes storage and execution boundaries, not player-facing behavior.

2. Requirements to Preserve

Area Current Requirement Redesign Treatment
Unit definitions Load unit metadata, stats, building flags, skills, carrier settings, race, help text, movement, vision, formation, and price. Keep Store as immutable UnitDefStore indexed by UnitDefId.
Skill definitions Load projectile/heal behavior, movement type, easing, slow effects, ray rendering, icons, damage kind, range, speed, homing, swing parameters. Keep Store as immutable SkillDefStore indexed by SkillDefId.
Battle runtime Update unit movement, targeting, skill execution, hit processing, buildings, resources, selection, and task progress. Change Move mutable fields into runtime stores processed by systems.
Build menu Load build actions with costs, requirements, build time, result type, spawn target, and move behavior. Keep Store as BuildActionDefStore; runtime build queues reference action IDs.
Resources Track gold, trust, food, resource points, icons, names, and amounts. Keep Definitions are immutable; runtime amounts are stored per faction/player.
Map data Represent battle tiles, buildings, unit placements, resource points, front/defense/neutral positions. Change Store tiles in flat arrays with tile-index addressing instead of nested objects.
UI/menu rules Visible menu controls should use plain Rect-based custom UI when visibility is an issue. Keep UI reads view models built from DOD stores; UI state stays separate from simulation state.
Data files Existing JSON files load units, skills, build menus, and resources. Change Keep JSON compatibility first; prefer TOML for new LoganToga2 configuration/data files.

3. Current Design Pressure Points

Large Mixed Objects

Unit contains definition data, runtime battle state, UI state, build state, movement state, strings, copied skill objects, and component pointers. Hot loops touch only a few fields but pull large cache lines.

Definition Copies

Units copy full Skill objects into arrSkill. Runtime instances should usually reference shared immutable skill definitions by ID.

Pointer-heavy Components

Array<std::unique_ptr<UnitComponent>> enables polymorphism, but it fragments memory and complicates deterministic runtime snapshots.

Nested Map Data

Array<Array<MapDetail>> is expressive, but battle systems usually need fast tile indexing, occupancy lookups, and compact flags.

Core Principles

  • Separate definitions from instances: content loaded from disk is immutable and referenced by compact IDs.
  • Use structure-of-arrays for hot runtime data: positions, HP, orders, factions, and movement states live in separate dense arrays.
  • Use handles instead of raw pointers: UnitId, SkillDefId, BuildActionDefId, and TileIndex are stable references.
  • Systems own behavior: movement, targeting, cooldowns, projectile simulation, building, and resource collection operate on stores.
  • UI is a projection: selection panels and menus consume read-only snapshots/view models, not mutable simulation objects.

5.2 Immutable Definition Stores

Store Data Notes
UnitDefStore Name tags, display names, image names, race, base stats, price, formation, building metadata, vision, maintain range, carrier capacity, skill IDs. Replaces repeated definition fields inside runtime Unit.
SkillDefStore Skill type, move type, easing, range, speed, damage kind, projectile size, icons, image keys, slow effects, ray settings, swing arc/reach. Skills are looked up by SkillDefId; texture assets register during load.
BuildActionDefStore Build action ID, display text, icon, costs, requirements, build time, category, spawn result, movement requirement. Runtime build queues store action IDs and progress only.
ResourceDefStore Resource name, icon, kind, initial amount. Player/faction amounts live in runtime resource arrays.

5.3 Runtime Unit Stores

struct UnitRuntimeStore {
    Array<UnitDefId> defId;
    Array<BattleWhichIsThePlayer> faction;
    Array<bool> alive;
    Array<bool> battleEnabled;

    Array<Vec2> positionLeft;
    Array<Vec2> orderPositionLeft;
    Array<Vec2> velocity;
    Array<moveState> movementState;

    Array<int32> hp;
    Array<int32> hpMax;
    Array<int32> mp;
    Array<int32> attack;
    Array<int32> defense;
    Array<int32> magic;
    Array<int32> magDef;
    Array<double> speed;
    Array<double> move;

    Array<UnitTask> task;
    Array<double> taskProgressSec;
    Array<BattleFormation> formation;
};

The runtime store keeps the fields updated every frame in dense arrays. Cold data such as localized names, help text, image names, and race strings remains in definition stores.

5.4 Optional Feature Stores

Feature Current Shape DOD Shape
Carrier CarrierComponent inside a unit component list. CarrierStore with carrierUnitIds, capacities, and stored unit ID lists.
Build queue Array<BuildAction> on each unit. BuildQueueStore keyed by builder unit ID, storing action ID, target tile, count, and progress.
Selection/UI Flags such as IsSelect, pressedUnit, temporary button rectangles. SelectionStore and UI-only view models. Simulation systems do not depend on UI flags.
Cooldown/uses Global maps and helper functions in battle code. CooldownStore and SkillUseStore indexed by (UnitId, SkillDefId).

Flat Battle Map

struct BattleMapStore {
    int32 width = 0;
    int32 height = 0;
    Array<String> tileTip;
    Array<uint32> flags;
    Array<ResourceDefId> resourceDefId;
    Array<int32> resourceAmount;
    Array<BattleWhichIsThePlayer> owner;
    Array<UnitId> occupyingBuilding;

    TileIndex index(int32 row, int32 col) const {
        return static_cast<TileIndex>(row * width + col);
    }
};

This replaces nested map access with a single index. It also makes fog, resource collection, pathing, building placement, and occupancy checks cheaper and easier to test.

7. System Pipeline

  1. Input System
    Reads keyboard/mouse/game commands and writes intent data: selected units, target positions, build requests, and skill requests.
  2. Order System
    Validates commands against definitions, resources, faction ownership, cooldowns, map constraints, and current tasks.
  3. Movement System
    Updates pathing state, velocity, position, arrival flags, formation offsets, and maintain range behavior using dense position arrays.
  4. Targeting System
    Scans candidate unit IDs by faction/spatial buckets and chooses targets according to skill definitions and battle rules.
  5. Skill Activation System
    Checks MP, cooldowns, uses, slow effects, and creates projectile or heal commands using SkillDefId.
  6. Projectile System
    Updates projectile trajectories, ray/swing behavior, homing, collision checks, and hit interval state.
  7. Damage/Heal System
    Applies skill effects to HP/MP arrays using explicit damage models for attack, magic, heal, and special cases.
  8. Build and Resource Systems
    Advance build queues, spawn units/buildings, collect resources, and update faction resource stores.
  9. Render View Builder
    Builds draw lists from runtime stores and immutable definitions. Rendering remains Siv3D-specific, while simulation remains data-centric.

8. Loader Redesign

The current Init(CommonConfig&) function should become a set of focused loaders. Each loader parses one content domain, validates references, and fills immutable definition stores.

Current Responsibility New Loader Output
Load Unit.json and copy skill objects into units. UnitDefLoader UnitDefStore with skill name references resolved to SkillDefId.
Load skill.json and register textures. SkillDefLoader SkillDefStore plus texture asset registrations.
Load build menu aaa.json. BuildActionDefLoader BuildActionDefStore and category/group indexes.
Load resource aaa.json. ResourceDefLoader ResourceDefStore and initial resource templates.
For LoganToga2, prefer TOML for new configuration/data files. The migration can support both existing JSON and new TOML by converting both into the same definition-store format.

9. Compatibility Strategy

Adapter Layer

Keep legacy Unit construction temporarily through adapters that build old objects from DOD stores. This allows incremental migration of screens and tests.

Save Migration

Preserve existing GameData fields. Add a versioned save section for battle world snapshots only when runtime battle persistence is introduced.

Command Line

Keep -Battle, -Battle001, and -test. Each entry point creates the same GameData, definition stores, and battle world as needed.

10. Migration Plan

  1. Create ID and Definition Types
    Add UnitDefId, SkillDefId, definition structs, and stores. Do not change battle behavior yet.
  2. Split Loaders from Main.cpp
    Move unit, skill, build action, and resource parsing into separate loader files. Validate missing keys and unresolved skill references.
  3. Replace Copied Skills with IDs
    Store skill IDs on unit definitions. Add helper lookup methods for UI and battle code.
  4. Introduce BattleWorld
    Spawn runtime units from definitions into dense arrays. Keep legacy battle code bridged through adapter accessors during transition.
  5. Migrate Hot Systems First
    Move movement, HP/MP update, cooldowns, skill uses, projectiles, and targeting into systems operating on stores.
  6. Flatten Battle Map
    Convert ClassMapBattle to BattleMapStore with a compatibility reader for existing map authoring data.
  7. Separate UI State
    Move selection, pressed unit, temporary command rectangles, and menu state out of simulation data. Continue using plain Rect-based controls where needed.
  8. Remove Legacy Object Dependencies
    Once systems and screens read from stores, reduce Unit to a compatibility shell or delete it if no longer needed.

11. Testing and Validation

Test Type Purpose Examples
Loader tests Ensure old JSON and new TOML produce identical definition stores. Unit count, skill references, build costs, resource kinds, default values.
Golden data tests Prevent behavior drift during migration. Known unit stats, known skill damage, known build result.
System tests Validate pure systems without rendering. Movement arrival, cooldown commit, carrier store/release, resource collection.
Scenario smoke tests Confirm existing command-line modes still run. -Battle, -Battle001, -test.
Performance checks Confirm DOD migration improves hot loops. Units updated per frame, projectile collision time, memory allocation count.

12. Risks and Mitigations

Risk Impact Mitigation
Large rewrite destabilizes gameplay. High Migrate through adapters and verify with golden tests before removing legacy paths.
ID indirection makes debugging harder. Medium Add debug name lookup helpers and inspector views for IDs.
Definition/runtime split requires many call-site updates. Medium Start with stores and helper accessors, then migrate systems one at a time.
JSON to TOML migration can break existing content. Medium Support both formats until generated stores match exactly.
Flat map loses authoring readability. Medium Keep readable source files and compile/load them into flat runtime arrays.

14. Definition of Done

  • All current unit, skill, build menu, resource, map, carrier, and battle requirements load into definition/runtime stores.
  • Existing command-line modes continue to work.
  • Battle behavior remains equivalent for validated scenarios.
  • Hot battle loops operate on dense arrays rather than large mixed objects.
  • UI state is separate from simulation state.
  • New configuration/data additions prefer TOML while preserving old JSON compatibility during migration.
  • Unit/system tests cover loaders, movement, cooldowns, projectile hits, damage/healing, build queues, resources, and carrier behavior.
