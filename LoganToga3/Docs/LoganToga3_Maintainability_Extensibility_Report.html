LoganToga3 保守性・拡張性 改善レポート

LoganToga3 の App / BattleWorld / Systems / Data / UI を横断して、冗長または削除可能な処理、抽象化できるロジック、今後の拡張に備えた分離ポイントを整理した設計レビューです。

対象: LoganToga3 目的: maintainability / extensibility 方針: テンプレートより関数・小クラス・データ駆動を優先 形式: 静的 HTML

1. 要約

最優先

UI 入力・描画・Rect 定義に重複が多く、機能追加時の変更箇所が増えやすい状態です。特にツールバー、タブ、数値増減ボタン、パネル hover 判定は共通化効果が高いです。

中優先

資源コスト、TOML 配列読み書き、ファイルパス解決、enum 変換が複数箇所に散っています。データ操作の小さな共通関数へ寄せると安全性が上がります。

低リスク

既にヘッダ単位で機能分割されており、段階的に整理できます。大規模な継承階層やテンプレート化を行わなくても改善可能です。

2. 調査した主な領域

領域主なファイル観察
アプリ入口Main.cpp, App/AppUpdate.h, App/AppRender.h, App/AppUiState.hGaussian アドオン、入力、更新、描画の流れは単純。ただしデバッグ UI と定義再生成処理が App 層に直接置かれています。
戦闘ワールドBattleWorld/BattleWorldStores.h, BattleWorld/BattleWorldOps.h, Systems/*SoA 形式のストアは方向性が明確です。一方、ユニット追加時に複数ストアの同期が手作業で、漏れに弱い構造です。
データ定義Data/DefinitionStores.h, Data/UnitCatalog.h, Data/Loaders/*.hTOML ベースの設計はプロジェクト方針に合っています。文字列配列、エスケープ、パス解決に重複があります。
エディタ UIUI/MapEditor*.h, UI/BuildingEditor*.hRect ベース UI の方針は一貫していますが、ボタン定義・入力・描画が別々に増殖しやすい状態です。

3. 冗長または削除候補の処理

優先度対象内容推奨
P1 App/AppUiState.h のデバッグボタン HandleDebugClipboardCaptureButton、HandleDebugClipboardCaptureShortcut、HandleDebugNewGameButtons、描画関数で、表示条件・hover・busy・クリック処理が個別に書かれています。 「デバッグ用 Rect ボタン」を扱う小さな関数にまとめます。テンプレートではなく、DebugButtonSpec のような構造体と通常関数で十分です。
P1 UI/MapEditorToolbarInput.h と UI/MapEditorDrawBasePanelsCore.h ツールバーのボタン番号、ラベル、active 条件、クリック時処理が入力側と描画側に分散しています。 ボタン一覧を配列データ化し、入力と描画が同じ定義を読む形にします。処理はラムダで保持できます。
P1 UI/BuildingEditorInput.h タブ 0〜4、サイズ変更ボタン、軸方向ボタン、reset 処理で同じ if パターンが繰り返されています。 タブ切替はループ化、軸方向は Point 配列 + 「現在のタブに応じてどの setter を呼ぶか」を関数化します。
P2 Systems/BattleQueries.h と Systems/BattleBuildOrders.h 資源の検索、所持量確認、消費処理が複数箇所にあります。建築ラインでは合計コスト計算も別実装です。 BuildActionCost 風の通常 struct と CanPayCost / PayCost を作り、単体・複数配置の両方で使います。
P2 Data/UnitCatalog.h と Data/Loaders/BuildActionDefTomlHelpers.h TOML 文字列配列の読み書き、文字列エスケープ、TextWriter へ直接書く未使用風関数が重複しています。 Data/TomlTextUtils.h のような非テンプレート helper に寄せます。未使用の writer 直書き関数は参照確認後に削除候補です。
P2 Resolve*TomlPath 系 App 配下と repo 配下を順番に見る処理が複数あります。候補数や fallback の方針もファイルごとに揺れています。 ResolveFirstExistingPath(candidates) を追加し、各ローダーは候補リストだけ持つ形にします。
P3 App/AppLifecycle.h, BattleWorld/BattleWorld.h, MapEditor.h など 集約 include だけのヘッダが多いです。便利ですが依存関係を見えにくくし、ビルド範囲を広げる可能性があります。 即削除ではなく、編集頻度が高いファイルだけ直接 include へ寄せる検討をします。

4. 抽象化できるロジック

4.1 Rect ベース UI 部品

プロジェクト方針として SimpleGUI ではなく Rect ベースの UI を使うべきなので、抽象化も Rect を前提にします。

  • DrawEditorButton を中心に、hover・active・disabled を扱う共通描画関数へ拡張する。
  • クリック判定と描画を同じ ButtonSpec データから作る。
  • タブ、close ボタン、+/- ボタンは専用の小関数へ分ける。

テンプレート不要。struct と通常関数、必要ならラムダで実現できます。

4.2 エディタのパネル hover 判定

DrawMapEditorCurrentCellMarker では多数の Rect を OR でつないでいます。新パネル追加時に漏れやすい構造です。

  • CollectBlockingEditorPanelRects(editor) を作る。
  • 各パネルが「カーソルをブロックする Rect」を返すように分離する。
  • 描画と入力で同じブロック判定を使えるようにする。

4.3 戦闘の資源コスト

現在は gold/trust/food が個別メンバーとして広がっています。資源種類が増えると修正箇所が多くなります。

  • BuildActionCost { gold, trust, food } を導入する。
  • ScaleCost(cost, count)、CanPayCost(world, defs, faction, cost)、PayCost(...) に寄せる。
  • UI 表示・保存は既存フィールドを維持し、内部処理から段階的に置き換える。

4.4 TOML ユーティリティ

UnitCatalog と BuildActionDefTomlHelpers に似た処理があります。

  • 文字列エスケープ、文字列配列読み込み、文字列配列生成を共通化する。
  • 「行ごとの自動改行を避ける」という既存方針に合わせ、全文を String として組み立てる関数を優先する。
  • writer へ直接配列を書く関数は、参照がなければ削除候補にする。

5. 戦闘システム設計の改善ポイント

対象問題改善案
SoA ストア同期 AddUnitToBattleWorld が units、cooldowns、buildQueues、carriers、pathing を順番に add しています。新ストア追加時に呼び忘れやすいです。 BattleWorldUnitStores::addUnitAll 相当の関数へ寄せるか、AddUnitToBattleWorld の直下に「追加必須ストア一覧」をコメントではなくコード上の配列的な構造で表す。
タスク遷移 Idle、Moving、Gathering、Building、Attacking への変更が各 system に散っています。 SetUnitTaskIdle、SetUnitTaskMoving のような副作用込み関数を作り、target / attackTarget / resourceTarget / path clear をまとめる。
更新順 UpdateBattleSimulation に順序が直書きされています。読みやすい一方、追加 system の意図が残りにくいです。 当面は直書きでよいですが、各行の目的を短い関数名に寄せる。テンプレートや system registry はまだ不要です。
配置判定 EvaluateBuildPlacementCell が resource / reservation / unit / object を直接調べています。 IsCellBlockedByResource、IsCellBlockedByUnit、IsCellBlockedByObject に分けると、将来の地形制限や陣営制限を足しやすくなります。

6. 削除前に確認すべき候補

以下は「すぐ削除」ではなく、参照検索・実行確認後の削除候補です。

  • WriteTomlStringArray / WriteTomlStringArrayValue: 現在の保存処理は文字列を組み立てて writer.write する方針に寄っているため、未使用なら削除候補です。
  • BuildCommandBaseFromCatalog: 読み込み処理内で直接使われていない場合、旧設計の残骸である可能性があります。
  • 集約 include だけのヘッダ: 公開 API として必要なもの以外は、将来の依存整理時に削減候補です。
  • App/example 配下の Siv3D サンプル資産: 配布物や実行時に不要なら、プロジェクトから除外または別管理候補です。ただし Siv3D の標準配置に依存している可能性があるため慎重に確認します。

7. 推奨リファクタリング順序

順序作業理由リスク
1Rect ボタン描画と入力の小さな共通関数化UI の重複が最も多く、効果が見えやすい。低
2BuildingEditor のタブ・数値増減・軸オフセット処理をループ化同一パターンの if が多く、バグ混入を減らせる。低〜中
3TOML 文字列ユーティリティを共通化保存形式の一貫性を保ちやすい。中。改行方針を必ず維持する。
4資源コスト処理を CanPay / Pay へ統合建築、ライン建築、将来の AI 消費で再利用できる。中。ゲームバランスに影響するためテスト必須。
5ユニットタスク遷移の関数化移動・採集・建築・攻撃の副作用を集約できる。中。既存挙動との差分確認が必要。
6未使用 helper と不要資産の削除最後に行うと、必要な helper を誤削除しにくい。低〜中。参照検索と起動確認が必要。

8. テンプレートを避けた抽象化方針

このコードベースでは、テンプレートを使わずに十分な抽象化が可能です。主に次の 4 つを使うのが安全です。

  • 小さな struct: ボタン、タブ、コスト、パネル Rect のようなデータをまとめる。
  • 通常関数: DrawButton、HandleButtonClick、CanPayCost のように明確な名前で共通化する。
  • ラムダ: その関数内だけで使う一時的な処理、例えば「クリックされたらこの値を変える」に使う。
  • データ駆動テーブル: ツールバーやタブのように「同じ形の項目が並ぶ」ものは配列で管理する。

9. 結論

LoganToga3 は、データ指向の BattleWorld と Rect ベース UI という方向性が既に見えています。現在の主な課題は、機能追加のたびに同じような入力・描画・保存・資源処理を各所へ足している点です。

最初に UI ボタンと BuildingEditor の重複を削ると、見た目や操作感を変えずに保守性を改善できます。その後、TOML ユーティリティと資源コスト処理を整理すると、ユニット・建築・資源の拡張がしやすくなります。
